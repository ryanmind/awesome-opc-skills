# Thinking Toolkit

一个通用的思维模型总入口 skill，适合用来做决策、拆问题、找根因、梳理定位、结构化表达，以及系统优化分析。

## 默认原则

- **结论优先**：先回答“应该怎么判断”，再解释理论
- **自动路由**：优先选最合适的 1～3 个模型，而不是堆很多概念
- **实战导向**：分析最后要落到动作建议
- **不强套模型**：如果某个模型没有帮助，就不要硬用
- **允许组合使用**：不同阶段的问题可以用不同模型接力解决

## 什么时候用

当用户在做下面这些事时，优先使用这个 skill：

- 在多个选项之间做选择
- 判断取舍、机会成本、是否继续投入
- 寻找问题根因，而不是只看表面现象
- 梳理职业方向、个人定位、IP 定位、战略聚焦
- 把混乱的问题讲清楚、拆清楚
- 识别系统中的关键杠杆点、反馈回路、长期复利点
- 把模糊想法整理成结构化答案、观点或内容骨架

常见触发语气包括但不限于：

- “帮我分析一下这件事”
- “我应该选 A 还是 B”
- “为什么我一直卡在这里”
- “帮我判断这个方向值不值得做”
- “帮我梳理一下我的定位”
- “这个问题的根因是什么”
- “帮我搭一个思考框架”
- “用几个思维模型拆一下”

## 什么时候不要用

下面这些场景不适合优先使用这个 skill：

- 用户主要是要查事实、查资料、查最新信息
- 用户只想润色文案、翻译、改措辞
- 用户只是想情绪支持，而不是做分析判断
- 已经有更明确、更专用的 skill 更匹配

## 默认输出顺序

默认按下面顺序输出：

1. **结论**
2. **为什么这么判断**
3. **使用的模型**
4. **下一步行动**
5. **如有必要：给模板 / 清单 / 表格**

除非用户明确要展开，否则默认保持简洁。

## 模型选择规则

- 从用户真正要解决的问题出发，不要只看用户提到的模型名
- 一般只用 **1～3 个模型**
- 如果既要诊断又要决策，优先一个模型查原因、一个模型做判断
- 如果两个模型有张力，要明确说出冲突点，而不是硬拼
- 问题简单时，一个好模型比五个弱模型更有用

## 读取指南

按需读取，不要一次性全加载：

- 想看某个模型的适用场景、误用提醒、输出结构，就打开对应的 `references/*.md`
- 想复用回答格式，就打开 `assets/output-templates.md`
- 除非真的要做横向比较，否则不要把所有 reference 一次性读进来

## 通用回答流程

默认按这个流程回答：

1. 先识别用户真正的问题或决策点
2. 用一句话给出结论
3. 说明最关键的 2～3 个原因
4. 点明用了哪些模型，以及为什么适合
5. 给出可执行的下一步动作
6. 如果用户要复用，再补模板、清单或表格

## 反模式

避免下面这些问题：

- 只罗列一堆模型，却没有帮用户解决问题
- 只讲概念，不转成行动建议
- 每次都硬套完整框架，导致回答臃肿
- 把思维模型当成绝对真理，而不是观察问题的镜头
- 因为模型“看起来高级”，就忽略具体上下文
